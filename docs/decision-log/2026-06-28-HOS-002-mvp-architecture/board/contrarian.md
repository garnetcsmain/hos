# Contrarian Review: HOS-2026-002

## My Recommendation (per decision)
- D1: **RESHAPE** - The rule engine is sound and explainable, but the hard-negative penalties are too weak to stop the exact failure mode they exist to stop (common-name false positive that sends a family to the wrong morgue), and "PII minimized before external call" is false as built.
- D2: **GREEN LIGHT** - Correct call for cold-start; the only real flaw is a hand-rolled `BEGIN/COMMIT` transaction wrapper on a global singleton that is not concurrency-safe and not actually rollback-isolated. Fixable without touching the decision.
- D3: **RESHAPE (borderline KILL)** - Fail-open auth on life-safety endpoints is a foreseeable catastrophe, not a "dev convenience"; the shared-token model also destroys the per-actor accountability that an append-only audit log is supposed to provide.
- D4: **RESHAPE** - The honesty framing is good but undercut by its own code: the in-app notification is hardcoded `status: "sent"` and an event `family.notified` is appended even though nothing reached the family. That is the exact false-delivery claim D4 promises never to make.

## Key Findings

### F1 — D1 — Hard-negative penalties are too weak to stop the named failure mode. Severity: CRITICAL
The penalties exist specifically to keep "two different people who share a common name" below the floor (engine.ts line 122-125). They do not.
- Scenario: A real "José García," male, age 30, last seen Maracaibo, is missing. A deceased male, "José García," age 31, processed in a Maracaibo morgue, is the found report. Name ~1.0, location high, sex match (+1), age diff 1 (~0.92), no contradiction penalty. Score lands 90+. Correct-looking — but it may be a *different* José García, one of dozens. Nothing in the engine models name *commonness*; a perfect match on the most common name in Venezuela scores identically to a perfect match on a rare one. The engine has no notion of base rate, so its confidence is systematically overstated for exactly the names most likely to collide.
- Worse, the contradiction penalty caps out at ×0.35 (sex) / ×0.4 (age>30). A name+location+trait match can score ~85 baseline; ×0.4 = 34, *below* the 45 floor — good. But sex conflict alone is ×0.35: an 85 baseline → 29, suppressed. Yet a borderline 60 baseline with a 19-year age gap (×0.65) → 39, also suppressed — fine. The dangerous case is the *common-name, no-contradiction* match above, where there is no penalty at all because nothing contradicts. The penalty system only fires on contradictions; it is silent on coincidence, which is the actual false-positive driver.
- *If I'm wrong:* if the coordinator UI surfaces "N other open reports share this name" alongside the score, the human absorbs the base-rate problem and this drops to HIGH. I did not find that signal anywhere in `views.ts` or the factor chain.

### F2 — D1 — "PII minimized before any external call" is false as implemented. Severity: CRITICAL (trust/legal)
The proposal and success criteria both assert PII is minimized before any external model call. The code sends the person's **full legal name verbatim** to Anthropic/OpenAI. `toPairInput` (aiAugment.ts line 22-47) ships `missing.fullName` and `found.fullName` — not the redacted `givenName` the public projection uses. It also ships the free-text `description` (up to 2000 chars), which is precisely where a relative writes "tiene una cicatriz de cesárea, diabético, mi número es +58 412…" The schema has a *separate* `sensitiveNotes` field that is correctly withheld — but families do not reliably segregate sensitive content into the right box; they pour it into `description`.
- Net effect: full name + city + last-seen area + free-text health/contact details of vulnerable people (some fleeing persecution) are POSTed to a US cloud provider the moment a key is set. The only thing "minimized" is `reporterContact` and `sensitiveNotes`. Calling this "PII minimized" in a board proposal is the kind of trust-layer violation the product principles forbid.
- *If I'm wrong:* if the deployment contract with the AI provider includes zero-retention/no-train terms and a DPA, the legal exposure shrinks — but the proposal claims minimization *technically happened*, and it did not. The claim itself is the defect.

### F3 — D3 — Fail-open auth is a foreseeable life-safety breach, not a dev convenience. Severity: CRITICAL
`assertCoordinator` returns successfully when `HOS_COORDINATOR_TOKEN` is unset (auth.ts line 16-22). Every privileged endpoint — `/api/matches`, `/api/matches/[id]` (full unredacted reports incl. `reporterContact`, `foundLocation`, `sensitiveNotes`), `/verify`, `/timeline`, `/notifications` — is then world-open. A `console.warn` once per process is the only guardrail.
- This is fail-*open*, and the failure is silent. The standard deployment accident — env var not propagated to the prod environment, typo in the var name, secret store miss — does not crash the app or 500; it serves full PII and lets anonymous callers confirm reunifications. In a crisis deploy done fast under pressure (exactly this product's operating condition), this *will* happen.
- The right default is fail-*closed*: no token configured → privileged endpoints 503. Make the dev-open path require an explicit `HOS_DEV_OPEN=1`, so "open" is a thing someone typed, never a thing they forgot.
- *If I'm wrong:* if deployment is gated behind a platform-level auth proxy (Vercel password / Cloudflare Access) that is mandatory and independently verified, the app-level fail-open is masked. But nothing in the proposal commits to that, and "defense in depth" means the app must not assume it.

### F4 — D3 — A single shared token makes the audit log lie about who acted. Severity: HIGH
D3 gates verify/notify behind ONE shared secret. But the audit story (a core principle) attributes every confirmation to `coordinator:${verifierOrg}` (verification.ts line 71), where `verifierOrg` is a free-text field the caller types into the request body (schemas.ts line 65). With one shared token, anyone holding it can self-attribute as any org. The append-only event log is meticulous — and forgeable at the actor level. "Auditability" that records an attacker-supplied actor string is theater. A wrong confirmation (family sent to wrong shelter/morgue) cannot be traced to a person, only to a self-declared org name.
- *If I'm wrong:* if Phase 0 is genuinely single-operator (one trusted coordinator, one org), per-actor attribution is moot and this is LOW. But the schema already models multiple `verifierOrg`s and the proposal lists "no org/tenant isolation between responding agencies" as a known con — so multi-org is the intended near-term state, and the audit gap is real.

### F5 — D4 — The system claims a delivery it did not make. Severity: HIGH
D4's whole premise is "never claim something happened that didn't." Yet on confirmation the code constructs the notification with `status: "sent"` hardcoded (verification.ts line 33) and appends an event `type: "family.notified"` with `actor: "system:notifier"` (line 113). Nothing was sent anywhere a family will see — there is no family-facing in-app inbox in this build; the only reader of `listNotifications()` is the *coordinator* console (notifications/route.ts is coordinator-gated). So "in-app notification, status sent, family.notified" describes a message the family has no way to read, recorded as delivered.
- This is the precise harm D4 names — a false sense that "the family was told" — implemented in the data model. A coordinator scanning the queue sees green "sent" rows and reasonably believes outreach occurred. It did not.
- Fix: status should be `queued` (it is queued, not delivered), and the event should be `notification.queued`, not `family.notified`. `family.notified` should only ever be written by a channel that actually reached the family.
- *If I'm wrong:* if there is a planned family-auth'd in-app inbox where the recipient genuinely reads this, "sent" is defensible for that channel. I found no such surface; the recipient field is even redacted (`•••`), which only makes sense for coordinator display, not delivery.

### F6 — D3/privacy — Public search enables re-identification of vulnerable people. Severity: HIGH
`/api/search` is open by design and matches on `fullName` + `city` (search.ts line 22-28), returning that a missing/found person exists with their given name, age band, city, and status. For a population that includes people fleeing persecution, an abuser, or a state actor, "is X listed as found, and in which city" is exactly the query you must not answer to an anonymous caller. Open intake is correct (families must report freely); open *enumeration by name* is a different decision that got bundled in without its own scrutiny. A found-person's city + status leaking to whoever they were hiding from is a direct physical-safety harm.
- *If I'm wrong:* if Phase 0's threat model explicitly excludes targeted-violence adversaries (pure natural-disaster context, no persecution), this is acceptable for speed. The Venezuela framing makes that exclusion hard to justify.

### F7 — D2 — The transaction wrapper is not isolated and not concurrency-safe. Severity: MEDIUM
`transaction()` does `db.exec("BEGIN")` on a single global connection (client.ts line 37-47). Three problems: (1) nested calls — `augmentCandidates` opens a transaction (aiAugment.ts line 70) while it can be reached from intake which already ran a transaction; SQLite has no nested `BEGIN`, so a nested call throws "cannot start a transaction within a transaction" and the inner rollback then throws "no transaction is active," masking the original error. (2) `BEGIN` (deferred) does not actually take a write lock until first write, so the "atomic" claim is weaker than stated under concurrent writers. (3) WAL + `busy_timeout` mitigates multi-*process* contention but a single shared mutable connection across Next.js request concurrency means interleaved `BEGIN/COMMIT` from concurrent requests can commit each other's partial work. This is a correctness bug independent of whether SQLite-vs-Postgres is the right call.
- *If I'm wrong:* if `node:sqlite` serializes all `exec` calls on the singleton and Next.js node runtime never runs two of these handlers truly concurrently in one process, (3) is moot — but (1), the nested-transaction throw, is reachable from the AI path today and I'd want a test proving intake→augment doesn't nest.

### F8 — D1 — Age similarity is brittle for the youngest, who matter most. Severity: MEDIUM
Age proximity uses `1 - diff/12` (engine.ts line 78): a 6-year gap scores 0.5. For adults that is reasonable; for a missing 4-year-old vs a found 9-year-old it treats them as a coin-flip, and reported child ages in crises are frequently estimates ("about 5"). Meanwhile the >30y *contradiction* penalty never protects a child case (the gaps are small). Children are the highest-stakes reunifications and the age model is tuned for adults.
- *If I'm wrong:* if the coordinator always eyeballs age and the engine is purely a pre-filter, the linear curve is fine. But the score is presented as confidence, and an overstated child-match score is high-harm.

### F9 — D1 — Nickname expansion is one-directional and unit-tested into a false sense of coverage. Severity: LOW
`expandName` maps nickname→canonical but `che`→`jose` collides with Argentine "che" (interjection/Guevara) and `rosa`→`rosario` will wrongly fuse a woman named Rosa with one named Rosario. These are asserted as features. The table is hand-built, Iberian-Spanish-leaning, and unvalidated against Venezuelan usage — a confident-looking source of both false merges (rosa/rosario) and silent misses. Low severity because it only nudges name sub-scores, but it's an unaudited correctness surface presented as "nickname-aware."

### Edge cases that bite
- **No-name found report:** `fullName` is optional for found reports (schemas.ts line 48). Name factor returns null and is dropped; the score then renormalizes over the *remaining* factors (engine.ts line 170 divides by `weightTotal`). So an unidentified body matched only on city+age+sex can score high *with no name evidence at all*, because the missing name doesn't penalize — it just vanishes from the denominator. A 100% "match" can mean "we agree on city and that it's a male adult."
- **Single-factor inflation:** if only one factor is assessable (say only sex is known on the found side), `base = score/weight = the factor's raw score`. Sex match alone → base 1.0 → score 100 before penalties. A coordinator sees "100" and a one-line evidence chain. The score's denominator-renormalization makes sparse data look *more* confident, not less.
- **Timeline floor:** `time` factor floors at 0.3 even when "found before last seen" by 1-3 days (engine.ts line 110-112) and only drops to 0.15 beyond -3 days. A logically impossible timeline still contributes positive signal.
- **Rate limiter trusts `x-forwarded-for`:** `clientKey` reads the client-supplied `x-forwarded-for` first (rateLimit.ts line 16). An abuser spoofs it and gets unlimited intake/search; meanwhile a whole NAT'd region behind one IP shares one bucket and locks each other out. In-memory windows also reset on every redeploy/scale event.
- **AI blend swing:** with one provider, blend is 50/50 (blend.ts line 25). A baseline-suppressed pair (sex conflict, baseline 29) where the cloud model hallucinates 0.9 → blended = 0.5×29 + 0.5×90 = 60 → **above floor, candidate surfaces.** The AI can *resurrect* a pair the deterministic hard-negative deliberately killed, and it does so at equal weight. The "baseline is always authority" philosophy in the blend comment is contradicted by the math.
