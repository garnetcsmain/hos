# Researcher Review: HOS-2026-002

## My Recommendation (per decision)
- D1: **RESHAPE** - Right call to abandon the phantom "5000-match ML model" and ship a deterministic explainable engine (this is exactly what ICRC/Primero use), but the Spanish two-surname normalization is broken and common-name false positives can clear the floor — both are crisis-grade harms.
- D2: **GREEN LIGHT** - node:sqlite behind a repository layer matches the real lesson from RapidFTR/Primero (offline-first, sync-when-connected); the single-node ceiling is a non-issue at Phase-0 volume and the migration path is real.
- D3: **RESHAPE** - A shared coordinator token is acceptable for Phase 0, but the dev-open-when-unset default is a deploy-time landmine, and "no per-user accountability" contradicts the audit-log principle for the most consequential actions (verify/notify).
- D4: **GREEN LIGHT** - In-app-only with explicit "not delivered externally" honesty is the single most defensible decision in the package and is directly vindicated by every comparable system; just don't let it ship as the *permanent* answer (Refunite's whole thesis is that SMS/USSD is the reach).

## Key Findings

**F1 — D1 — Spanish two-surname normalization systematically under-scores likely-true matches.** Severity: **HIGH**.
`assignmentScore` in `similarity.ts` divides the token-pairing total by `Math.max(tokensA.length, tokensB.length)`. I ran the actual algorithm: "Jose Garcia" vs "Jose Garcia Perez" scores **67%** name similarity, and "Maria Jose Rodriguez Lopez" vs "Maria Rodriguez" scores **50%**. This is not an edge case — it is *the* dominant Spanish-name data-entry failure mode. Per US patents on Hispanic-name data quality and Spanish naming-customs literature, when a form has one last-name field "the second apellido may disappear entirely"; a relative abroad routinely gives one apellido while a shelter logs both (or vice versa). At name weight 0.34, that 33% loss strips ~11 points off the overall score and can drop a real family below MATCH_FLOOR=45. *If I'm wrong:* if intake forms enforce structured given/paternal/maternal fields (they don't today — names are free-text `fullName`), or if real Venezuelan intake data shows surname counts are usually symmetric, this drops to LOW. (Flagged as a spin-off task.)

**F2 — D1 — Common-name false positives can clear the floor; the contradiction penalty only fires on sex/age, never on surname disagreement.** Severity: **HIGH**.
"Jose Garcia" vs "Jose Martinez" — different people — scores **50%** on name alone (shared given name, `tokenStrength` floors the surname mismatch to 0 but the matched given name still yields 0.5/2 tokens... actually 1 matched token / 2 = 0.5). With plausibly co-located age-matching records and no sex/age conflict, the pair can exceed 45. The hard-negative penalties (`contradictionPenalty`) punish sex mismatch (x0.35) and age gap (x0.4) but there is **no penalty for an actively contradictory surname**. In a crisis this is the exact failure that "sends a grieving family to the wrong morgue." Google Person Finder's own post-mortems emphasize *verification to combat misinformation* precisely because coincidental-identity noise is rampant at scale. *If I'm wrong:* the human verifier is the backstop and TOP_K=5 keeps the queue small, so a false candidate is "merely" reviewed and rejected — but risk F6 (volunteer over-trust) is real and undercuts that defense.

**F3 — D3 — Dev-open-when-token-unset is fail-open on the most dangerous endpoints.** Severity: **HIGH**.
`auth.ts` returns early (grants access) when `HOS_COORDINATOR_TOKEN` is unset, logging one warning. The endpoints thus exposed are verify/notify/timeline/full-PII — the ones that resolve cases and reveal contact info. Fail-open is the wrong default for a system whose threat model includes deployment by non-experts in a hurry. RapidFTR/Primero's explicit security posture is the inverse: "limits access to trained, registered users" via encryption and access control as a *precondition*, not a convenience-off default. *If I'm wrong:* if deploy tooling hard-fails the build/boot when the token is unset in any non-local NODE_ENV, the landmine is defused and this drops to LOW. Right now nothing enforces that.

**F4 — D3 — A single shared secret gives no per-actor accountability, which undercuts the audit log.** Severity: **MEDIUM**.
The event store records `actor: coordinator:${verifierOrg}` — but `verifierOrg`/`verifierName` are *self-reported in the request body* (`verification.ts`), not derived from an authenticated identity. Anyone with the one shared token can attribute a confirmation to any org/name. The append-only log is honest about *what* happened but not *who*, which is a real gap for the highest-stakes action in the system (confirming a reunification). *If I'm wrong:* if Phase 0 is genuinely single-operator (one coordinator), per-user identity is premature and this is LOW.

**F5 — D1 — The AI blend sends full free-text `description` to third parties; "PII minimized" is overstated.** Severity: **MEDIUM**.
`toPairInput`/`prompt.ts` exclude contact and precise location (good), but pass the entire `description` free-text field verbatim to Anthropic/OpenAI. Descriptions in real intake routinely contain incidental PII (medical conditions, tattoos, "last seen with her brother Luis at the Petare clinic"). The comment claims "no medical notes" but nothing redacts them — the field is just trusted to be clean. ICRC's Trace-the-Face model deliberately "flags a match without accessing whole databases"; data minimization there is structural, not advisory. *If I'm wrong:* if the description field is product-constrained to non-PII traits only (it isn't — it's free text), this is LOW. Also note the 50/50 blend lets one external model move a score 50 points; that is a large authority handoff for a "force multiplier."

**F6 — D1/D4 — No learning loop + volunteer over-trust is a documented, not hypothetical, risk.** Severity: **MEDIUM**.
The proposal itself lists "volunteers may over-trust ranked output." Every comparable system converged on *human verification as a hard gate* (Primero: "the case worker needs to then do the tracing and subsequent verification"; Trace-the-Face: "thorough verification process"). HOS gets this right architecturally (D4, verification service). The gap is that confirmed/rejected verifications are logged but never fed back to tune weights — so the hand-tuned weights stay unvalidated against real outcomes indefinitely. *If I'm wrong:* at cold start there's nothing to learn from yet, so this is correctly deferred — it only becomes HIGH once volume exists and the loop still isn't built.

**F7 — D2 — SQLite single-node write concurrency is a real but correctly-deferred ceiling.** Severity: **LOW**.
WAL + busy_timeout handles single-node concurrent reads/writes fine; the limit is multi-instance writers (multiple agencies on separate nodes). At Phase-0 single-deployment volume this won't bind, and the repository layer makes Postgres a drop-in. This is the *right* offline-first choice — it mirrors RapidFTR's "saved automatically and uploaded when network access becomes available" model, which is the proven pattern for low-connectivity crisis data capture.

**Ground-truth on Spanish name matching:** Spanish naming uses given name + paternal apellido + maternal apellido ([Spanish naming customs](https://en.wikipedia.org/wiki/Spanish_naming_customs)). The documented record-linkage failure is that one-field forms drop or merge the second apellido ([dos apellidos](https://history.denverlibrary.org/news/genealogy/dos-apellidos-when-families-have-two-surnames)); commercial systems (multiple US patents on Hispanic-name data quality) handle this with structural segment-aware processors and percentage-match scoring rather than bag-of-tokens divided by the longer list. HOS's `expandName` nickname map (Pepe→José) is genuinely good and ahead of naive matchers — but the surname-count normalization (F1) is the known trap, and I confirmed it numerically against the real code.

**Ground-truth on false-positive harm:** every system above makes verification a *mandatory human step* and none auto-confirm — converging evidence that the "AI recommends, people decide" principle is correct and that false hope is treated as real harm industry-wide. The risk in HOS is not the architecture (correct) but the scoring permissiveness (F2) combined with deferred per-user accountability (F4).

## Confidence Score
**0.74.** High confidence on the architecture-vs-industry comparison and on the two name-matching findings (F1/F2 confirmed by executing the actual algorithm, not opinion). Docked because: (a) I could not find published false-positive *rates* for Trace-the-Face/Primero to quantify how bad F2 is in practice — the search returned no accuracy statistics; (b) I did not exhaustively trace every API route to confirm which endpoints `assertCoordinator` actually guards (I confirmed the gate's fail-open logic and rate-limit wiring, but not 100% endpoint coverage); (c) I have no real Venezuelan intake sample to confirm how often surname counts are asymmetric (the premise of F1's severity).

## What I Don't Know
- Real false-positive/false-negative rates of the comparable systems — no public numbers found; my severity ranking on F2 is reasoned, not measured.
- The actual distribution of Venezuelan-2026 intake data (surname symmetry, how often age/sex/location are populated). The whole engine weighting (name 0.34, location 0.20) is unvalidated against it; with sparse fields, "weighted only over assessable factors" means a name-only match can dominate.
- Whether deploy/boot tooling fails closed when `HOS_COORDINATOR_TOKEN` is unset in production — decisive for F3's real severity; I did not find such a guard.
- Whether the `description` free-text field is product-constrained anywhere upstream (validation `schemas.ts` exists but I did not confirm it strips PII) — decisive for F5.
- Legal/data-protection regime for sending even minimized Venezuelan PII to US cloud AI providers (Anthropic/OpenAI) — out of my lane but a likely blocker the proposal doesn't address.
- Whether HOS intends a PFIF-style interop export; absence is fine for Phase 0 but matters if it must federate with other responders later.
