# Expansionist Review: HOS-2026-002

## My Recommendation (per decision)
- **D1: GREEN LIGHT** - The deterministic-engine-with-pluggable-AI split is not a cold-start compromise, it is the actual moat: every confirmed/rejected verification is a labeled training row, so the rule engine plus event log *is* the data flywheel HOS-2026-001 assumed someone else would hand you.
- **D2: GREEN LIGHT** - The repository layer turns "SQLite vs Postgres" into a non-decision and unlocks an edge/offline-first topology (tablet in a shelter, sync later) that hosted-Postgres-only competitors structurally cannot match in low-connectivity crises.
- **D3: RESHAPE** - Keep the token gate now, but the single shared secret caps you at one operator; the moment a second agency wants in, the absence of an org/actor identity field becomes the bottleneck on the entire network-effect thesis. Add an `actor`/`org` dimension to events *now* (cheap), defer real authN.
- **D4: RESHAPE** - In-app-only is the honest floor, but the `channel` column and queued-notification model are already a provider-agnostic outbox; greenlight the abstraction, and treat the notification queue as the seam where HOS becomes a multi-channel reach layer, not just a matcher.

## Key Findings

**F1 — D1 — The confirmed/rejected verifications are unlabeled training data being thrown on the floor. [HIGH]**
Evidence: `verification.ts` writes `match.confirmed` / `match.rejected` events with the candidate id, and `matcher.ts` logs the full factor vector + score at suggestion time. That means for every verified candidate you already have `(factor_vector, model_version, human_label)` recoverable by joining two event types in `events`. HOS-2026-001 assumed 5000 historical matches had to exist before any learning could start; this architecture *manufactures* that dataset as a byproduct of normal operation — every coordinator decision is a row. The 10x is: ship the rule engine to N deployments, and in 6 months you have the only labeled crisis-reunification corpus in existence, region-stratified, with provenance. *If I'm wrong:* if verification volume stays in the low hundreds for years, this is a learning loop with no fuel and the rule engine's hand-tuned weights are forever the ceiling.

**F2 — D1 — The `AiProvider` interface is a clean ensemble socket, not just a "blend a cloud LLM" hook. [MEDIUM]**
Evidence: `types.ts` defines `AiProvider.scorePair(input): Promise<MatchSignal>` and `blend.ts` already averages over *N* usable signals with per-provider weight. Nothing in that contract assumes "LLM." A future facial-recognition model (HOS-2026-006, already scoped), a phonetic-match service, a biometric vendor, or a *second NGO's own model* all satisfy the same interface and drop into the same audited blend. The architecture is quietly a multi-signal matching marketplace. *If I'm wrong:* if real deployments only ever run the baseline (no keys, no budget), the socket is unused weight and the 50/50 blend is dead code.

**F3 — D2 — The repository + event-store pair is the substrate for cross-deployment federation, the largest upside in the whole package. [HIGH]**
Evidence: all access is behind `repositories/*`, and the event store is append-only with `entityType`/`entityId`. Append-only event logs are the canonical primitive for replication and CRDT-style sync. A Venezuela deployment and a Colombia deployment running the same schema can, later, exchange *redacted candidate-pair events* across a border without sharing raw PII — exactly the "cross-border search" that scope.out defers pending legal review. The repository layer means the SQLite→Postgres→federated-mesh path is one adapter swap, not three rewrites. The moat: refugee flows are inherently cross-border, and a system that can match a missing report in Caracas against a found report in Cúcuta is categorically more valuable than any single-jurisdiction tool. *If I'm wrong:* if legal/data-sovereignty review kills cross-border sharing permanently, federation is dead and SQLite is merely "adequate single-node storage."

**F4 — D3 — The single shared token is the one true bottleneck on the network-effect thesis. [HIGH]**
Evidence: `auth.ts` compares one `HOS_COORDINATOR_TOKEN`; events record `actor` as a free-text string like `coordinator:${verifierOrg}` (verification.ts line 71) but there is no org entity, no per-org token, no isolation. The whole expansionist case (F1, F3) depends on *multiple agencies* contributing reports and verifications. The instant agency #2 joins, one shared secret means no per-org accountability, no revocation, no tenant boundary — and worse, the labeled-data corpus (F1) loses its most valuable column: *who* labeled it and whether they were reliable. *If I'm wrong:* if Phase 0 is genuinely single-operator forever (one Red Cross instance, never shared), the token is fine and org identity is premature.

**F5 — D4 — The notification queue is an outbox pattern that generalizes to a reach/delivery-receipt layer. [MEDIUM]**
Evidence: `notifications` table has a `channel` column (currently `"in_app"`) and a `status` column (`"sent"`), and `family.notified` events log `{ channel }`. This is structurally an outbox: add an SMS/WhatsApp adapter and the same row gets `channel:"whatsapp"`, `status:"queued"→"delivered"→"read"`, each transition an audited event. The upside: HOS becomes the *system of record for "did the family actually find out,"* with per-channel delivery receipts — a trust artifact NGOs and donors will pay for. *If I'm wrong:* if external providers are never wired (no budget, no provider relationships), the channel column stays a one-valued enum and this is over-engineering for a feature that never ships.

## Biggest Upside + Domino Effects + Moat

**The biggest upside: HOS is accidentally a data-generation machine for the exact dataset HOS-2026-001 said it lacked.**

The deviation that looks like a retreat ("we have no ML model, no 5000 historical matches") is the strongest strategic move in the package. By shipping a deterministic engine whose every output is a structured factor vector, and gating every confirmation behind a logged human verification, the system converts *operation* into *labeling*. You don't buy a training set; you grow one, with provenance, region tags, and the human rationale attached. Nobody else has this corpus because nobody else has run an explainable matcher across real crisis intake with a human-in-the-loop log. That is the moat — not the rule weights (copyable in an afternoon), but the **proprietary, auditable, region-stratified labeled corpus** that the architecture mints as exhaust.

**Domino chain (each unlocks the next):**

1. **Rule engine + event log** → labeled corpus accrues (F1). No new infra; it's already being written.
2. **Labeled corpus + pluggable `AiProvider` socket** (F2) → train a *learned* signal that plugs into the *same* blend the cloud LLM uses, validated against held-out human labels. The rule engine's evidence chain becomes the explainability baseline a black-box model must beat, so you get learning *without* losing auditability — the thing HOS-2026-001 couldn't promise.
3. **Repository layer + append-only events** (F3) → redacted-event federation across deployments → cross-border matching, the single highest-value capability for refugee flows, reachable as an adapter not a rewrite.
4. **Federation + org identity** (requires fixing F4) → a *network*: each agency's reports raise the match rate for every other agency. Classic two-sided network effect — more found-reports make every missing-report more likely to resolve, and vice versa. The Nth agency to join gets more value than the first, which is the structural reason a latecomer competitor can't catch up.
5. **Notification outbox** (F5) → delivery-receipt layer → the auditable "the family was actually reached" record that turns HOS from a matching tool into the **trust/accountability system of record** donors fund and governments cite.

**The moat, stated plainly:** explainable-by-construction + human-in-the-loop + append-only audit produces a *defensible labeled dataset and a cross-agency network*, both of which compound and neither of which a better-funded competitor can buy or rebuild quickly. The architecture's "honesty layer" (no false claims, full audit) is not just ethics — it's the credential that lets multiple distrustful agencies share one network, which is the precondition for the network effect.

**The resource bottleneck to capture this upside:** the binding constraint is **not** any of the deferred external services (cloud key, Postgres, SMS provider — all are adapter-swaps the architecture already anticipates). It is **multi-org identity (D3)**. Every compounding asset above — corpus provenance, per-labeler reliability, tenant-safe federation, delivery accountability — requires a first-class `org`/`actor` entity. Right now that identity exists only as a free-text string in an event payload. The cheap, high-leverage move is to model `org` and `actor` as real columns/entities *now*, while the data is small, and defer real authN/OIDC until a provider exists. If org identity is not seeded into the schema before the corpus grows, you will be retrofitting tenancy onto a dataset whose most valuable dimension — *who decided, and were they right before* — was never captured. That retrofit is the one that gets structurally harder every day the system runs, and it is the throat of the whole expansion.

## Confidence Score
**0.7.** High conviction that the architecture *enables* every domino above — I verified the seams exist in code (clean `AiProvider` interface, append-only `events` with recoverable label joins, repository indirection, channel/status outbox). Not higher because the expansionist thesis is entirely demand-conditional: the corpus, network effect, and federation moat are all worthless without *adoption by multiple agencies and real intake volume*, and I have zero evidence of demand, partner pipeline, or even a second operator. The technology is necessary but radically insufficient; a beautiful network-effect substrate with one user is just a database. The single biggest discount is F4 — if multi-org never happens, three of my five dominoes (corpus provenance, network effect, federation) collapse to single-tenant, and the upside shrinks from "category-defining moat" to "good explainable matcher."

## What I Don't Know
- **Demand-side reality:** Is there a pipeline of NGOs/agencies who would actually join a shared network, or is Phase 0 destined to be one operator? The entire network-effect case lives or dies here and the proposal says nothing about partners.
- **Legal ceiling on federation (F3):** scope.out defers cross-border to legal review. If data-sovereignty law forbids even redacted-event exchange across jurisdictions, the highest-value domino is dead on arrival regardless of architecture.
- **Verification volume:** How many human verifications per month does a realistic deployment generate? At 50/month the learning loop (F1) is starved for years; at 5000/month it's a flywheel. The proposal gives no throughput estimate.
- **Whether the factor vector is actually recoverable for training:** I confirmed `matcher.ts` logs factors+score and `verification.ts` logs the label, but I did not verify the candidate's *factor snapshot at decision time* is immutably preserved if the candidate is later re-scored (e.g. AI augmentation in `aiAugment.ts` overwrites `candidate.factors`). If re-scoring clobbers the snapshot the coordinator actually saw, the label is attached to the wrong features and the corpus is poisoned — worth a hard check before claiming the data moat.
- **Cost model for the cloud-AI blend:** scoring every candidate pair against an external LLM (`aiAugment.ts` loops per candidate) could be expensive at scale; I don't know the per-match economics or whether the 50/50 blend is even the right weighting once a learned signal exists.
