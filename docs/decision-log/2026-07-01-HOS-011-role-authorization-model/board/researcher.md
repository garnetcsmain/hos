# Researcher Review: HOS-2026-011 (role and authorization model)

*Define agency-admin's scope, place the remote/digital contributor, and set the org-isolation boundary.*
Reviewer: Researcher Agent · Date: 2026-07-01 · Method: live web research (Primero/UNICEF, Ushahidi, SBTF/MicroMappers, GitHub, multi-tenant RBAC + Postgres RLS guidance) + direct read of the HOS schema, auth gate, and the HOS-2026-001-08 task file to verify the "current state" claims rather than trust them. Prior-round context on the remote/field distinction (from HOS-2026-010-R1's `board/r1-researcher.md`) was re-verified against how the platforms model **permissions**, not just how they recruit — that is a different question and it changes one thing (see Q3).

> **Scope note.** My job here is "what does the world actually look like," not to cast a build/no-build vote. Two of the board's five open questions have strong, convergent real-world answers (agency-admin's shape; where remote contributors go). Two are decided more by HOS's *own* schema and fail-closed precedent than by any comparable (isolation enforcement layer; PII-table coverage). One (the mid-tier site-lead role) has clear precedent both for and against, and is genuinely a judgment call I can inform but not settle.

## My Recommendation

Adopt the **Primero-shaped model**, because Primero is not an analogy — it is the same problem (multiple aid agencies sharing one child-protection case-management instance, each seeing only their own beneficiaries' PII) solved and deployed at national scale by UNICEF. Concretely:

1. **agency-admin is genuinely distinct from coordinator, on the org-isolation axis, and it is the more important of the two roles to get right.** Model it exactly as Primero's *agency administrator*: full people/settings authority **scoped to its own org only** (manage its own volunteers, its own sites/needs/offers, its own cases), with **zero visibility into any other org's data**. Model `coordinator` as Primero's *superuser* / cross-org tier: the cross-org response visibility HOS's coordination hub needs. The split is **breadth-of-data (own-org vs all-orgs), not function** — that is precisely how Primero separates them, and it maps cleanly onto HOS's existing `org_id`-scoped coordination tables.

2. **Remote/digital contributors are NOT a fourth top-level role and NOT merely a flag on responder — the cleaner model is: "responder" is field-shaped, and the remote contributor is a distinct member type whose permissions attach to a *task function* (translate, moderate), not to a geography.** This is how SBTF/MicroMappers actually structure remote work (functional teams — Media Monitoring, Geolocation, Verification — with permissions per function). But because the prior board already settled that geofencing is **action-scoped, not role-scoped** (HOS-2026-010-R1-D2), the "don't inherit field assumptions" problem the proposal worries about is *already solved by construction*: a translator never triggers a geo check because a translation edit structurally carries no location claim, regardless of what role they hold. So the remote-vs-field distinction should live where the prior board put it — in what the *action* is — and the role only needs to grant the right *task permissions*, not encode "field" vs "remote." Practically: add remote contributors as a low-privilege member type (task-permissioned, self-registering, org-optional), and do **not** reuse the `responder` role's field-shaped defaults for them.

3. **Org isolation MUST be DB-enforced (Postgres RLS), not app-layer filtering — this is not a close call given HOS's own fail-closed precedent.** Every comparable and every best-practice source lands on "enforce the tenant boundary at the database so a buggy query cannot leak across tenants." For a project whose stated principle is "when in doubt, refuse" (Board D3, and the fail-closed auth gate I read in `auth.ts`), app-layer-only isolation — where a single missing `WHERE org_id = …` silently exposes another org's family-tracing PII — is the exact failure mode fail-closed exists to forbid. RLS is defense-in-depth; app filtering alone is defense-in-hope.

4. **Load-bearing schema finding the proposal's open questions do not surface:** HOS's `org_id` isolation column exists on the **coordination** tables (`sites`, `needs`, `offers`) but **NOT** on the **case/PII** tables (`missing_reports`, `found_reports`, `verifications`) — those carry only a free-text `reporter_org` / `verifier_org` *provenance string*, which cannot back an RLS policy. So "an agency-admin can only see their own org's PII" is **not implementable on the most sensitive data without a schema change** (adding a real `org_id` FK to the case tables) that HOS-2026-001-08 has not scoped. The role model this proposal defines should state explicitly whether case/PII data is even org-partitioned or is a shared cross-org pool — because the current schema quietly assumes the latter, and that is a policy decision, not an implementation detail.

## Comparable Systems

- **Primero / UNICEF CPIMS+ (child-protection case management, multi-agency, national deployments — the closest possible comparable):** Roles are built from two **orthogonal** axes: a *permission* axis (read / write / export / manage, per resource — Cases, Incidents, Reports, Users, Agencies) and a *"Management"* (data-breadth) axis with the literal options **"Access only my records or user," "Access all records or users in my group," "Access all records or users,"** and **"No records — Admin functionality only."** Built-in tiers: **Superuser** ("manage" on everything, incl. Agencies + System Settings = HOS's cross-org coordinator); **System Administrator** (a role with **no record access** — "meant for users who act solely as system administrators… without being able to access cases"); **Agency Administrator** (read/write users + **"agency read"** — "view and edit user accounts only within their own agency"); **Manager** (group-scoped — approves case plans, sees "all records in my group"); **Caseworker** ("my records" only). → Directly answers three of HOS's questions: agency-admin is a real, distinct, **own-org-only** role; there IS a proven **manager tier between org-admin and caseworker**; and the clean way to express all of it is *permission × data-breadth*, not a flat role list. Notably Primero does **not** distinguish remote vs field in its permission model — that split isn't a permission concern there. → [Primero Admin Guide](https://support.primero.org/pdfs/primeroIMS/PrimeroAdminGuide/master/book.pdf) · [Primero CPIMS User Guide](https://github.com/primeroIMS/PrimeroCPIMSUserGuide/blob/master/UsersGuide.md) · [CPIMS+](https://www.cpims.org/)
- **Ushahidi (crisis mapping, used by many partner orgs):** Default **Admin** = "full control over ALL functionality on your deployment"; default **Member** = "edit their own posts" only; custom roles configurable in between. **Crucially, a Ushahidi *deployment* is effectively single-tenant** — everyone in one deployment shares one dataset. The sector's way of running "multiple orgs" on Ushahidi has historically been **one deployment per org**, not in-deployment tenant isolation. → A real alternative to HOS's plan worth naming: "separate instance per agency" sidesteps the whole org-isolation-RBAC problem, at the cost of the cross-org coordination visibility that is HOS-2026-007's entire reason to exist. HOS has *chosen* the harder shared-instance path; the comparison shows that is a deliberate, non-free choice. → [Ushahidi Roles](https://docs.ushahidi.com/ushahidi-platform-user-manual/4.-managing-people-on-your-deployment/4.1-roles)
- **Standby Task Force / MicroMappers / Digital Humanitarian Network (remote-volunteer backbone of the sector):** Structure remote volunteers into **functional teams** — Media Monitoring, Geolocation, Verification, Report, Task — and **permissions attach to the function**: "Media Monitoring… volunteers creating reports but not approving, verifying, or geolocating them"; the Verification team is the one that "may mark reports as verified." The entire platform is **remote-by-construction with no field-access role at all** — remote isn't a *variant* of a field role, it's the whole population, differentiated by *what task you're trusted to do*. → This is the sharpest evidence for HOS's Q3: in the systems built around remote contributors, the meaningful role axis is **task function** (create vs verify vs geolocate), not "remote vs field." HOS should permission its remote translator/moderator by *task*, not by geography. → [SBTF structure](https://standbytaskforce.org/about-us/our-structure/) · [SBTF 101](https://www.standbytaskforce.org/2011/08/28/standby-task-force-101/)
- **Translators Without Borders / CLEAR Global (100k+ remote volunteers — the exact translator cohort):** Self-register, no local voucher, and the org is **moving away from the verified/non-verified distinction** entirely — most work "claimable by anyone regardless of status." → Confirms the remote contributor is a **major real population**, self-registers, and that hard verification gates for this cohort cut *against* the sector trend. (Re-verified from HOS-2026-010-R1; unchanged.) → [CLEAR Global / TWB](https://clearglobal.org/translators-without-borders/)
- **GitHub organizations (mature, widely-copied org-RBAC reference):** **Owner** (full org admin) / **Member** (default, non-admin) / **Outside collaborator** (access to specific repos but *not a member of the org* — "a consultant or temporary employee") / **Team maintainer** (elevated *within one team*, not org-wide). Repo roles ladder Read → Triage → Write → Maintain → Admin, where **Maintain** = "project managers who need to manage the repository without access to sensitive or destructive actions." → Two transferable ideas: (a) the **outside-collaborator** pattern is a near-perfect mental model for the remote translator — contributes to specific work without being a full org member; (b) **"Maintain" and "Team maintainer"** are exactly the "site-lead between admin and member, minus destructive powers" tier HOS's Q5 asks about, and GitHub found it worth a distinct role. → [GitHub org roles](https://docs.github.com/en/organizations/managing-peoples-access-to-your-organization-with-roles/roles-in-an-organization)
- **General multi-tenant SaaS RBAC guidance (WorkOS, LoginRadius, Auth0):** Consensus: roles are **always tenant-scoped, never global** ("a user who is admin in Workspace A should have zero elevated permissions in Workspace B"); the primitives are *permission (action:resource) → role → role-assignment scoped to a tenant*; and enforce the boundary **"at schema (tenant_id everywhere), runtime (resource pre-checks), and tests (cross-tenant access attempts must fail)"** — "enforced twice, once structurally, once at runtime." → Independent confirmation that HOS's `org_id`-on-every-row shape is the right substrate, and that the acceptance test must include *a cross-org access attempt that is asserted to fail* — not just a positive-path test. → [WorkOS](https://workos.com/blog/how-to-design-multi-tenant-rbac-saas) · [LoginRadius](https://www.loginradius.com/blog/engineering/rbac-saas-multi-tenant-b2b-platforms) · [Auth0](https://auth0.com/blog/how-to-choose-the-right-authorization-model-for-your-multi-tenant-saas-application/)

## Ground-Truth Data

- **Primero "Management" (data-breadth) scope levels:** exactly **four** — my-records / my-group / all-records / admin-only-no-records — and they are set **independently** of the read/write/export permissions. (source: Primero admin guide, quoted above)
- **Primero agency isolation mechanism:** the **"agency read"** permission — an agency admin can "view and edit user accounts only within their own agency"; cross-agency movement of a *case* happens through explicit **referral/transfer**, i.e. isolation is the default and crossing it is a deliberate, logged action. (source: Primero admin guide)
- **RLS vs schema-per-tenant, for HOS's actual profile:** guidance is consistent — **RLS wins** when tenants are "uniform in size and behavior," schema changes are "frequent," the team is small, and tenant count is not in the thousands; **schema-per-tenant** ("maximum isolation… high operational overhead… migrations across potentially hundreds of schemas") is overkill below that scale. HOS = a handful of agencies, one small volunteer-run ops team, an evolving schema → **RLS is the indicated choice**, not separate schemas/DBs. (source: Propelius, WellAlly, Supabase RLS guides)
- **Supabase-specific RLS gotcha that intersects HOS's fail-closed principle:** put the org/tenant claim in **`raw_app_meta_data`** (server-controlled), **never `user_metadata`** — "this information can be modified by authenticated end users," so an RLS policy keyed on user-editable metadata lets a user rewrite their own org_id and read another tenant's rows. This is a concrete, easy-to-get-wrong fail-*open* trap directly in HOS's chosen stack. (source: [Supabase RLS docs](https://supabase.com/docs/guides/database/postgres/row-level-security))
- **Remote-volunteer population size (sanity check that this role is worth defining):** TWB alone reports **100,000+** self-registering remote volunteers; SBTF/DHN federate many more. Not an edge case. (source: CLEAR Global; SBTF)
- **HOS schema, verified this session (not from the proposal's own summary):** `orgs` table exists (`schema.pg.ts` L103) with `id/name/kind` and **no membership/role join table anywhere** (grep for `membership|user_org|role TEXT` across `apps/web/app/lib/db/` returns nothing — the entire authz layer is greenfield). `sites`/`needs`/`offers` carry a real `org_id` **FK** (+ `idx_sites_org`); `missing_reports`/`found_reports`/`verifications` carry only free-text `reporter_org`/`verifier_org` **provenance**, no `org_id` FK. `events.actor` is free **text**, not a typed role/org actor. Coordinator gate in `auth.ts` is explicitly **fail-closed** (503 when unconfigured). (source: repo read, 2026-07-01)

## My Proposed Role Model

Grounded in Primero (isolation + tiers), SBTF (remote-by-task), GitHub (outside-collaborator + maintainer), and multi-tenant RBAC consensus (per-tenant roles, enforce-at-every-layer):

**Express roles as two orthogonal axes, not a flat list** (Primero's proven shape):
- **Data-breadth axis (who you can see):** `own-org` | `all-orgs` | `admin-only (no case data)`.
- **Capability axis (what you can do), per resource:** submit report · view coordination console · create/update site · verify a match · contact a family · edit/retract (per HOS-2026-010-R2).

Mapped to HOS's four named needs:

| Role | Data breadth | Capability surface (concrete HOS actions) | Isolation |
|---|---|---|---|
| **coordinator** (= Primero superuser) | **all-orgs** | full: view console across orgs, create/verify/contact, cross-org coordination; the existing cross-org hub role | sees all orgs by design; the *only* cross-org tier |
| **agency-admin** (= Primero agency admin) | **own-org only** | manage own org's responders + sites/needs/offers; view/verify/contact **within own org's cases only**; manage own-org settings/people | **RLS-enforced** to `org_id`; cannot query another org's rows, period |
| **responder** (field volunteer, HOS-2026-010 identity) | own-org (or org-optional) | submit found/missing reports; view assigned work; **no** admin/settings; geo applies **per-action**, not per-role | scoped to own org where org-affiliated |
| **remote/digital contributor** (translator, moderator — new, task-permissioned; NOT a `responder` flag) | task-scoped; org-**optional** (self-registers) | only the task functions granted (translate, moderate); **never** field/site/verify defaults unless explicitly granted; self-registers, no local voucher | if org-affiliated, RLS-scoped; if unaffiliated, sees only the task queue assigned, no org PII |
| *(optional, deferable)* **site-lead** (= Primero manager / GitHub maintainer) | own **site/group** within an org | manage just their site's responders/capacity; **not** org-wide settings, **not** destructive actions | group-scoped under the org's `org_id` |

**Isolation enforcement (non-negotiable, per HOS's own fail-closed rule):**
1. **RLS on Postgres**, keyed on an `org_id` claim carried in Supabase **`raw_app_meta_data`** (server-set), **never `user_metadata`** (user-editable → fail-open).
2. **Add a real `org_id` FK to the case/PII tables** (`missing_reports`, `found_reports`, `verifications`) — or make an explicit, documented board decision that case/tracing data is a **shared cross-org pool** and only *coordination* data (sites/needs/offers) is org-isolated. Do not leave this implicit; the current schema silently implies "shared."
3. **Add the missing membership primitive:** a `user ↔ org ↔ role` join (none exists today). Roles are **per-org assignments**, not a global column — a person can be agency-admin of org A and nothing in org B.
4. **Acceptance test must include a cross-org access attempt asserted to FAIL** (the RBAC-consensus "cross-tenant access must fail" test), not only a positive-path "can see my own" test — mirroring the D4 honesty precedent (prove the boundary holds, don't assume it).
5. Wire the signed-in identity into `events.actor` so the audit log attributes actions to a real actor+org, not a free-text string (currently `actor TEXT`).

**On the remote contributor specifically:** treat like GitHub's *outside collaborator* + SBTF's *task team* — a first-class but low-privilege member whose permissions are the **task** (translate/moderate), decoupled from geography. The geofencing-inheritance worry is already handled by action-scoping (HOS-2026-010-R1-D2); the remaining job is simply not to hand them field/responder defaults.

## Unknown Unknowns

- **Is case/tracing PII even supposed to be org-partitioned?** I can confirm the *coordination* tables are org-scoped and the *case* tables are not (schema fact). What I cannot tell from outside is whether that's an intentional "reunification data is a shared humanitarian pool, provenance-tagged" policy or an unfinished-schema accident. The entire meaning of "agency-admin sees only their org's PII" hinges on this, and it's a **policy** question for the principals/human, not a research finding. This is the single biggest fork the proposal leaves unstated.
- **Whether HOS actually wants cross-org visibility at all, or per-org instances would serve the mission better.** Ushahidi's real-world answer to "multiple orgs" is often *separate deployments*. If HOS's coordination hub value is smaller than assumed, the whole RBAC-isolation build is avoidable by running one instance per agency. I lean toward shared-instance being right (HOS-2026-007 is premised on cross-org coordination), but I'm inferring intent, not measuring it.
- **Real org count and whether roles will stay uniform.** RLS-vs-schema and fixed-vs-custom-roles both hinge on tenant count/heterogeneity. Every comparable's threshold ("thousands of tenants," "enterprise wants custom roles") is imported; HOS's actual number of responding agencies and whether any will demand bespoke roles is unmeasured. My RLS + fixed-role recommendation assumes "handful of agencies, uniform needs" — true today, unverified for the 2-year horizon.
- **Whether a translator ever legitimately needs to see PII to do the task.** A translator translating a *family message* may need the message content (which is PII), which complicates "remote contributor sees no org PII." I flag this as an unresolved interaction between the remote role and data-minimization that the capability matrix above hand-waves ("task queue") and that a real workflow walk-through, not research, must settle.
- **Migration/enforcement cost of retrofitting `org_id` + RLS onto an existing (if young) schema and its ~12 already-wired coordinator routes.** I can say RLS is the right *target*; I can't estimate the churn of getting there from the current all-coordinator-gated state without reading every route, which is outside this review's scope.

## Confidence Score

**0.72**

Why this high: the two questions the board most needs answered have a **near-ideal comparable**. Primero is not a loose analogy — it is UNICEF's deployed, multi-agency, child-protection case-management system solving *exactly* HOS's "multiple orgs, isolate each other's beneficiary PII, one shared instance" problem, and it independently converges with GitHub's org model, the SBTF/MicroMappers remote-task pattern, and the general multi-tenant RBAC consensus on the same answers: agency-admin = own-org-scoped and distinct from a cross-org superuser; a mid-tier manager role is real and precedented; remote contributors are permissioned by task, not geography; and tenant isolation must be DB-enforced. The isolation-layer recommendation (RLS, fail-closed) is over-determined by both the comparables *and* HOS's own D3 precedent, and I verified the enabling schema shape (and its gaps) in the repo directly rather than trusting the proposal's summary.

Why not higher: (1) the **single most decision-relevant fact — whether case/PII data is org-partitioned or a shared pool — is a policy choice HOS has not made and I cannot make for it**; my model works either way but the proposal can't close without that answer. (2) The Primero admin-guide PDF 404'd on direct fetch; my verbatim scope/permission strings come from a search snippet that quoted it plus the corroborating user guide, so the *exact* label wording (e.g. "agency read") is high-but-not-certain and should be confirmed against a live Primero admin guide before it's quoted as gospel in an implementation spec. (3) The mid-tier site-lead question has genuine precedent **both** ways (Primero/GitHub ship it; HOS functions without it today), so my "real but deferable" verdict is a judgment call, not a fact. (4) Every scale-dependent threshold (RLS-vs-schema, fixed-vs-custom roles) rests on "HOS = a handful of uniform agencies," which is true now but an assumption about the future, not a measurement.
