# Expansionist Review: HOS-2026-012

## My Recommendation
RESHAPE FOR MORE UPSIDE

The three asks in this proposal — volunteer skills, volunteer needs, donor actor type — are not three features. They're three sightings of the same animal from three different angles. Ship them as three separate bolt-ons and HOS ends up with three parallel person-shaped schemas that each half-solve "what can this human do / need / offer right now," none of which talk to `Need`/`Offer` (HOS-007) or to each other. Ship them as one **Participant** model — a person or org, with a set of typed capabilities (skills, resources, capacity) and typed needs, matched by the SAME engine that already matches orgs — and HOS has built the substrate for every human-to-task and human-to-human match it will ever need, volunteer/donor/beneficiary alike. This is the highest-leverage schema decision this project makes this cycle. Reshape for that, not for caution — the caution (location precision, honest three-option UX) is a real second axis, and I address it below, but it does not require shrinking the model.

## Biggest Upside
**A single participant graph where every person and org in the system — volunteer, donor, beneficiary, shelter, NGO — expresses "what I have" and "what I need" in one typed vocabulary, matched by one deterministic engine.** Today HOS has two disconnected worlds: `Org`/`Need`/`Offer` (things, HOS-007) and nothing at all for people (explicitly out of scope, per coordination.ts's own comment). This proposal is the first crack at the second world. Built narrow, it stays a second world forever — a "volunteer profile" bolted next to the org marketplace, never talking to it. Built as one model, HOS gets ONE marketplace: a shelter's `Need` for formula, a volunteer's `Offer` of driving-with-a-car, a donor's `Offer` of fuel, and — later — a beneficiary's `Need` for a ride to a clinic, are all the same kind of row, ranked by the same `rankOffersForNeed`-family function, shown through the same "advisory, human decides" UI pattern this project has now used twice (HOS-007, and again in spirit here).

- **Scale:** every human who ever touches HOS — not just the volunteers this proposal names, but the donors it names AND the beneficiaries it doesn't yet, because the substrate is identical.
- **Reach:** this is the difference between "HOS coordinates aid between organizations" and "HOS coordinates aid between everyone" — a categorically bigger mission surface, achieved by generalizing one type instead of adding three.
- **Timeline:** the unification is a modeling decision, not a build-out — it costs almost nothing extra NOW (a `subjectType: "org" | "person"` discriminator and a `subjectId`, instead of three bespoke shapes) and saves a full second matching engine, a full second UI pattern, and a future costly merge later. The volunteer-skill and donor slices proposed here can still ship first and alone; the ambition is in the shape they're cut to, not in scope creep on day one.

## Domino Effects
1. **Unlocks: `Need`/`Offer` generalized to `subjectType: org | person`** -> Enables: **one matching engine serves both markets.** `rankOffersForNeed` already does category + district + quantity scoring on `Offer`/`Need`. A volunteer's "I can drive, I have fuel" is structurally an `Offer` (category: `logistics`/`transport`, quantity: 1, unit: `trip` or `hours`) against a shelter's `Need` for a formula run — no new engine, the same one, extended with a person-aware scoring dimension (proximity-to-person instead of proximity-to-site). This is the single highest-leverage move in the whole proposal: it turns "build a second matching system" into "add one field and one scoring dimension to the first one."
2. **Unlocks: donor-as-Offer-source, no new actor type needed** -> Enables: **donors and orgs compete for the same visibility on equal footing.** A donor "I have a truck and fuel" and an NGO "we have a truck and fuel" are the same `Offer` row with a different `subjectType`. A coordinator sees one ranked list of "who can move formula to this shelter," not a volunteer list, then a donor list, then an org list, checked three times by hand. This is the domino that makes "donor" NOT a fourth parallel concept — it's the first proof that the unified model actually pays for itself immediately, on this very proposal's own third ask.
3. **Unlocks: a person's switchable skill = their currently-active `Offer` set** -> Enables: **the "driver today, cook tomorrow" requirement falls out of the model for free**, because `Offer.status` (`available`/`committed`/`depleted`) already exists (HOS-007) as the exact state machine a switchable skill needs: a volunteer has an `Offer(driving)` and an `Offer(cooking)`, and "which am I today" is just which one is `available` vs. temporarily withdrawn. No bespoke "current skill" field, no new state machine — reuse of a Board-approved primitive that already shipped.
4. **Unlocks: the same substrate expressed for beneficiaries** -> Enables: **HOS's actual mission-critical unaddressed gap** — a displaced family's `Need` for a ride, a translator, a wheelchair, a specific medicine — matched against the SAME volunteer/donor `Offer` pool this proposal is building. This proposal is explicitly volunteer/donor scoped and I am not asking to build beneficiary-facing matching now (that's a much bigger trust/safety/access surface — literacy, device access, exploitation risk from a bad match reaching a vulnerable person directly). But if the model is `Participant`-shaped from day one, that future is a new `subjectType` and a new capability-scope check (HOS-011's model already supports `scope: self`), not a rebuild. If it's volunteer/donor-shaped from day one, it's a second migration.

## Moat Opportunity
**The participant-matching standard: one typed vocabulary of "has" and "needs" that spans organizations, volunteers, donors, and — eventually — the people HOS exists to help.** Ushahidi, CrisisCleanup, and Team Rubicon-style volunteer-management tools each solved ONE side of this (org-to-org supply chains, OR volunteer-to-task dispatch) but not both in one coherent graph — that's the Researcher's question to verify with real precedent, but structurally it's the gap. If HOS is the system where a shelter's need, a volunteer's spare hour, and a donor's spare tank of gas are the same kind of object, ranked the same way, that's:

- **Data:** a participant graph — who has reliably offered what, matched to what, over time — that compounds exactly like the HOS-2026-010 Expansionist review's reporter-reliability graph, except now it covers EVERY human in the system, not just field reporters. Two compounding graphs (report-reliability + participant-reliability) sharing one identity substrate (HOS-2026-010's device-key+PIN) is a materially bigger moat than either alone.
- **Standard:** "advisory match, exactly N honest options, human decides" as a documented UX pattern for humanitarian dispatch generally — this proposal's "exactly three" idea, if validated, is itself a portable interaction standard other crisis tools could adopt, the same way HOS-007's deterministic-explainable-matcher pattern is already a repeatable in-house standard used twice.
- **Trust brand:** a coordinator who has used HOS to match orgs, volunteers, AND donors through one consistent, explainable, never-auto-committing interface has one mental model for the whole system, not three — that's a retention and word-of-mouth moat inside the exact under-resourced, high-turnover volunteer coordination teams this is built for.

## Adjacent Opportunities
- **Beneficiary-facing matching** (see Domino #4) — the largest one, deliberately not proposed for THIS decision, but the reason to shape the model right now instead of later.
- **Cross-crisis reuse.** A `Participant` with typed capabilities/needs is not Venezuela-specific or even disaster-specific — it's the shape of any mutual-aid network. Every future HOS deployment inherits it free.
- **Donor-to-donor and volunteer-to-volunteer matching**, not just donor/volunteer-to-need. A donor with a generator and no diesel, matched to a donor with diesel and no generator, is the same engine, zero new code, once the model is unified. This is a genuinely new mutual-aid pattern HOS hasn't discussed anywhere yet — a byproduct of generalizing rather than a designed feature.
- **The remote/digital contributor role from HOS-011** (task-scoped, org-optional — translation, moderation) is ALSO a `Participant` with an `Offer` (translation capacity) and no location dimension at all — proof the model already needs to support offers with zero location-matching weight, which is the same design constraint the Contrarian/Principals will raise about volunteer location precision. One model, one constraint, satisfied once.

## Where To Be Bolder, Not More Cautious
The proposal's own instinct — "exactly three, heavily location-weighted, human decides" — is right and should NOT be softened. Where I'd push harder:

- **Don't model "skill" and "personal need" as two new bespoke shapes.** Model them as `Offer` and `Need` with `subjectType: "person"` added to the existing HOS-007 types (today `orgId: string` on both — generalize to a `subjectId`/`subjectType` pair, a small, mechanical, non-breaking change). This is the single reshape that turns three asks into one and is buildable NOW, with nothing invented — every field these need (category, district, quantity/unit, status, urgency) already exists on `Need`/`Offer`.
- **Don't invent a fourth actor type for `donor` sitting outside `Org`.** An individual donor with no org affiliation is exactly the `subjectType: "person"` case above, with `orgId`/`subjectId` pointing at a person record instead of an `Org` row — structurally identical to how HOS-011 already settled that `field-volunteer` and `remote contributor` are `org-optional` capability bundles, not org-dependent ones. Reuse that precedent; don't re-litigate "does a person need an org" a second time in the same cycle.
- **Do build the location-weighted proximity scoring as a NEW scoring dimension on the SAME `rankOffersForNeed`-family function**, not a parallel matcher. HOS-007 already scores district-match as a bonus, not a filter — the honest, non-surveillance version of "location-weighted" this proposal wants is a proximity SCORE contributing to rank, computed from the same coarse `district` grain HOS-007 and HOS-010 already settled on, not a new precise-location primitive. This satisfies "must be location based to maximize speed" without inventing the individually-trackable data this project has twice rejected — ambition and honesty are the same design here, not a tradeoff.
- **Push the "exactly three, never padded" rule into the shared engine as a reusable primitive**, not a one-off UI rule for this feature: a `topN(matches, n, floor)` helper that returns fewer than N when fewer than N clear the usefulness floor, used by every future advisory-matching surface HOS ships. That's the moat's UX half — codify it once, reuse it everywhere, and it becomes as durable a pattern as the deterministic-explainable-matcher already is.

## Resource Bottleneck
**Secure the `Need`/`Offer` generalization to `subjectType: org | person` NOW, in this decision, even though full volunteer/donor build-out correctly waits on HOS-2026-010's device-key+PIN substrate landing.**

1. This is a schema decision, not a feature build — it costs nothing extra to specify correctly today (one discriminator field, one `subjectId` reference) versus specifying it narrowly and paying a real migration later to merge three shapes into one. The cost asymmetry is entirely in HOS's favor RIGHT NOW, while the data is still small — the exact "model it now while it's cheap" logic this project's own coordination.ts comment already uses to justify modeling `Org` early.
2. Ride HOS-2026-011's capability-scope model (`self`, `org:<id>`, `site:<id>`, now proven extensible) for the authorization half — a person's `Offer`/`Need` rows are `scope: self` resources, which that model already supports natively. Don't invent a second authorization idea for "who can see/edit a volunteer's need" when HOS-011 already solved "who can see/edit a self-scoped resource."
3. Do NOT let this decision wait on HOS-2026-010's identity substrate to be fully built before the MODEL is specified — the proposal itself notes the profile "has nothing to attach to until that substrate exists," which is true for BUILDING it, not for DESIGNING it. Specify the unified shape now so that when device-key+PIN lands, the volunteer/donor profile has one correct home to attach to instead of a redesign fight.
4. The thing NOT to rush: the beneficiary-facing extension of this model (Domino #4). That's real upside, correctly not in scope for this decision, and carries a materially different trust/safety/access bar (a vulnerable person being matched TO by strangers is a different risk profile than a volunteer choosing to help). Name it as a future direction in the decision record so the model is shaped to allow it later — but do not build or scope it now.

## Confidence Score
0.78 — high conviction that unifying `Need`/`Offer` across `subjectType: org | person` is both the biggest available upside AND the cheapest possible time to capture it (a field addition to code the Board already approved, not new infrastructure), and that it directly resolves two of the board's own open questions (is donor a new actor type; are volunteer needs a new `Need` instance) with the SAME answer. Not higher because: (a) I have not verified how much actual code outside `coordination.ts`/`match.ts` assumes `orgId` is always an `Org` foreign key (UI components, seed data, any coordinator dashboard rendering) — the generalization could ripple further than the two files I read; (b) "exactly three, location-weighted" as sound UX is the Researcher's and User's call, not mine, and my recommendation to keep it bold assumes it survives that scrutiny; (c) the beneficiary-facing domino is the biggest single upside I named and I'm explicitly NOT recommending building toward it yet, which is the right call but means the very biggest number in this review is a "later, if" rather than a "now."

## What I Don't Know
- Whether `orgId` is referenced as a hard foreign-key assumption anywhere outside `coordination.ts`/`match.ts` (UI rendering, seed data, any existing dashboard) that would make the `subjectType` generalization costlier than the two files I've verified suggest.
- Real precedent for whether volunteer-management systems that already do person-to-task matching (CrisisCleanup, Team Rubicon, NVOAD) ALSO unify that with their org-to-org supply chains in one model, or deliberately keep them separate for reasons I haven't seen — this bears directly on whether "one Participant model" is a genuine industry-leading move or something the field has tried and found not worth it. Researcher's territory.
- Actual volunteer/donor volume and density in a real Venezuelan deployment — the participant-reliability graph and the "donor-to-donor matching" adjacent opportunity are both functions of enough participants existing to make matching non-trivial, and I have no numbers, same gap the HOS-2026-010 Expansionist review already flagged for the reporter-reliability graph.
